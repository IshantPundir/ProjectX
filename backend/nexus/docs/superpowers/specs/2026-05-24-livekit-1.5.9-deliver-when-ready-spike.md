## 1. Suppress the auto-reply — `StopResponse`

`on_user_turn_completed(self, turn_ctx: ChatContext, new_message: ChatMessage)` is the hook called
**when the user's turn ends, before the agent's reply**. Raising `StopResponse()` from it aborts the
framework's automatic reply for that turn. Docs (push-to-talk example):

```python
async def on_user_turn_completed(self, turn_ctx, new_message) -> None:
    if not new_message.text_content:
        raise StopResponse()   # stop the agent from generating a reply
```

`StopResponse` is already imported and used in our `agent.py` (path (c) on barge-in, and in
`llm_node` when nothing is current). **Key fact for Phase 2:** after `StopResponse`, the
`AgentSession` stays alive and can still produce speech *explicitly* via `say()`/`generate_reply()` —
exactly as `on_enter` produces proactive opener speech with no preceding user turn. The per-activity
speech-scheduler task (`AgentActivity._scheduling_atask`) runs for the whole activity, not per-turn,
so callback-driven speech scheduled **later in the same committed turn** is honored.

⇒ Phase 2 shape: `on_user_turn_completed` launches triage ∥ brain tasks, registers their
done-callbacks, then `raise StopResponse()` and returns. Nothing is awaited in silence (D3).

## 2. Speak a filler from a done-callback — `session.say`

Installed 1.5.9 signature:

```python
def say(self, text: str | AsyncIterable[str], *,
        audio: NotGivenOr[AsyncIterable[rtc.AudioFrame]] = NOT_GIVEN,
        allow_interruptions: NotGivenOr[bool] = NOT_GIVEN,
        add_to_chat_ctx: bool = True) -> SpeechHandle
```

- Returns a `SpeechHandle` immediately (a Future-like). It is **not** a coroutine — calling it just
  *schedules* TTS; `await handle` is a shortcut for `handle.wait_for_playout()`. Our current code
  already calls `self.session.say(self._ack(), add_to_chat_ctx=False)` fire-and-forget.
- ⇒ Safe to call from an `asyncio.Task.add_done_callback(cb)` (the callback runs on the event loop).
- Use `add_to_chat_ctx=False` for the filler (don't pollute the chat context; we capture the
  transcript from reliable in-process sources, not the chat ctx — fix `3f29a082`).
- Requires a TTS plugin (we have one) — fine.

## 3. Deliver the real answer via the mouth — `session.generate_reply`

Installed 1.5.9 signature:

```python
def generate_reply(self, *, user_input=NOT_GIVEN, instructions=NOT_GIVEN,
                   tool_choice=NOT_GIVEN, tools=NOT_GIVEN, allow_interruptions=NOT_GIVEN,
                   chat_ctx=NOT_GIVEN, input_modality='text') -> SpeechHandle
```

- Also returns a `SpeechHandle` and is callable fire-and-forget. The docs' `on_enter` examples call
  `self.session.generate_reply(...)` **without await**.
- It routes through the active agent's `llm_node`. Our `_MouthAgent.llm_node` ignores the passed
  `chat_ctx` — it reads `controller.current_for_turn(self._current_turn_ref)`, builds its own
  `ChatContext.empty()` from `mouth.build_turn_messages(directive, …)`, and delegates to
  `Agent.default.llm_node`. So **`session.generate_reply()` with no args** is the correct call: the
  mouth voices whatever directive the controller has staged for the current turn.
- ⇒ Phase 2: the brain task's done-callback stages the brain's `Directive` on the controller, then
  calls `session.generate_reply()` (no await). The mouth's Pass-2 (now filler-aware via
  `_last_filler`) voices it, continuing from the filler triage spoke.

## 4. Ordering — filler then question, **no interruption** (the decisive finding)

Read from `AgentActivity._schedule_speech` (installed source): every `say`/`generate_reply` pushes its
handle onto a **priority max-heap** drained by the scheduler task:

```python
heapq.heappush(self._speech_q, (-priority, time.perf_counter_ns(), speech))
```

All call sites schedule at `SpeechHandle.SPEECH_PRIORITY_NORMAL` (= 5; `LOW`=0, `HIGH`=10). So:

- Equal priority ⇒ tie-broken by `perf_counter_ns()` ⇒ **FIFO by schedule time**.
- Scheduling a new speech **does not preempt** the one currently playing — it waits in the queue.
  Interruption happens only on user barge-in or an explicit `interrupt()`.

⇒ Timeline on a `TO_BRAIN` answer turn: triage filler is scheduled first (~0.8–2s) and plays; the
brain's `generate_reply` is scheduled later (~3s) and **enqueues behind the filler**, playing after it
finishes. If the filler already finished, the queue is empty and the question plays immediately. Either
way the candidate hears "filler … question" as one continuous turn. **No explicit await/coordination
between the two callbacks is required, and we must NOT call `interrupt()` between them.**

(Trailing-open filler punctuation — `…`/`—` — plus the filler-aware Pass-2 prompt is what makes the
two TTS utterances *sound* like one sentence; the queue guarantees the *order*.)

## 5. Barge-in / cancellation semantics

- **Playing speech:** user speech auto-interrupts the agent (adaptive interruption on LK Cloud; the
  agent stops and truncates history to the heard portion). `SpeechHandle.interrupted` reports it;
  `session.interrupt(force=…)` / `handle.interrupt()` force it. (1.5.7 added "pause output when user
  starts speaking during thinking"; 1.5.10 — *after* us — adds more cancellation hardening. We are on
  1.5.9.)
- **In-flight reasoning:** the existing `user_state_changed -> "speaking"` handler calls
  `agent.cancel_brain()`. Phase 2 extends it to **also cancel the in-flight triage task**.
- **HANDLED route** cancels the speculatively-launched brain task (accepted waste, §3 of the spec).
- **Done-callback safety (Phase 2 must implement):** a task done-callback must
  (1) no-op if `task.cancelled()` (barge-in / HANDLED-cancel), and
  (2) no-op if its turn is no longer current (stale/superseded) — guard on a turn-seq / the
  controller's staleness — before scheduling any speech. Otherwise a cancelled or stale turn could
  schedule an orphaned question after the user already moved on.

## 6. Resulting `on_user_turn_completed` shape (for the Phase 2 plan)

```
on_user_turn_completed(turn_ctx, new_message):
    append fragment to _pending_answer; capture transcript; turn bookkeeping
    accumulated = join(_pending_answer)
    launch triage task  (TriagePlane.triage on accumulated answer)         # nano clock
    launch brain  task  (ControlPlane.decide on accumulated answer)        # mini clock (speculative)
    triage_task.add_done_callback(_on_triage_done)   # say(filler); HANDLED -> cancel brain
    brain_task.add_done_callback(_on_brain_done)      # stage directive; generate_reply() (TO_BRAIN)
    set brain_pending=True (mute reflex cues for the window)
    raise StopResponse()                              # never await; deliver via callbacks
```

`_on_triage_done`: if cancelled/stale → no-op. Else `session.say(spoken_line, add_to_chat_ctx=False)`.
If `route==handled`: cancel the brain task; if `replay_last_question` → stage a `REPEAT` directive +
`generate_reply()`; else (still-pending) accumulate, no directive, clear `brain_pending`.
`_on_brain_done` (TO_BRAIN only): if cancelled/stale → no-op. Else supersede any speculative pre-stage,
stage the directive, store `_last_filler`, `session.generate_reply()`, clear `brain_pending`,
reset `_pending_answer` on advance/clarify/repeat.
