# Behavioral Layer + Intro/Warmup Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a behavioral verification phase (claim-verification gate before technical depth) and an intro/warmup turn (persona-anchored greeting + role brief) to the AI interview engine. Recruiter sees both kinds in one bank; engine processes them through the unchanged Judge/State/Speaker pipeline plus three small additions.

**Architecture:** Behavioral and technical questions live in the same `stage_question_banks` bank, differentiated only by the existing `question_kind` column (populated since migration 0026, unused by engine until now). The bank generator splits its single LLM call into two per-kind calls. The engine adds one new InstructionKind (`intro_brief`), one new flag (`is_post_phase_transition`), and pauses the lifecycle timer until the candidate first speaks. Frontend hides the engine boot latency behind a loading state. **No new Phase FSM, no Judge prompt rewrite, no State Engine invariant changes.**

**Tech Stack:** Python 3.13 + FastAPI + SQLAlchemy async (asyncpg) + Dramatiq + Alembic + OpenAI Responses API + LiveKit Agents SDK. Frontend: Next.js 16 + LiveKit React SDK.

**Spec:** `docs/superpowers/specs/2026-05-19-behavioral-layer-and-intro-design.md`

def _filter_behavioral_eligible(signals: list[dict]) -> list[dict]:
    """Return knockout signals voice-verifiable in the behavioral phase.

    The behavioral phase verifies claims (years, platforms, employer scope)
    via open-ended questions. We restrict to `experience` and `behavioral`
    signal types — credentials (degrees, certs) are pre-filtered by ATS,
    and competency signals are depth-flavored (technical phase territory).

    See docs/superpowers/specs/2026-05-19-behavioral-layer-and-intro-design.md §1.
    """
    return [
        s for s in signals
        if s.get("knockout") is True
        and s.get("type") in ("experience", "behavioral")
    ]
```

```
# YOUR TASK

Generate the BEHAVIORAL VERIFICATION questions for this stage's bank.

Behavioral verification confirms — via open-ended candidate self-report —
that the candidate actually has the knockout-required experience BEFORE
the technical scenarios run. Think of these as the "tell me about
yourself" beat tailored to verify the JD's knockout signals.

Generate 1 open-ended question by default. Generate a 2nd ONLY when the
knockout signals span unrelated domains (e.g., language requirement +
management experience) that one open-ended question cannot reasonably
cover via fan-out. Hard cap: 2 questions.

# OUTPUT SCHEMA

You return the same `StageQuestionBankOutput` schema described in the
common prompt. Critically:

- `question_kind` MUST be `"behavioral_star"` for every question.
- `is_mandatory` MUST be `true` for every question — they're the gate.
- `estimated_minutes` should be 1.5-2.5 per question (open-ended but
  not deep-dive).
- `signal_values` must reference the BEHAVIORAL-ELIGIBLE knockout
  signals only (the input signal list has already been filtered to
  these). One question can fan-out across multiple knockouts.

# QUESTION SHAPE

NOT "design a system that does X" (that's technical_depth territory).

YES "walk me through your background — total years, the iPaaS platforms
you've actually used in production, and where you've focused most."

The shape is CLAIM-VERIFICATION, not depth-extraction. The candidate
reports facts about their experience; the engine's Judge extracts
observations and verifies the knockout signals are at least at
`partial` coverage. Failure path: candidate explicitly discloses
no experience → existing knockout-policy fires → polite_close.

# POSITIVE EVIDENCE ANCHORS

Anchors should describe CLAIM SHAPES the candidate would state, NOT
technical specifics they would design. Examples:

  - "States total years in integration work (≥8 for sufficient)"
  - "Names MuleSoft / TIBCO / Boomi as a platform used in production"
  - "States years hands-on on the named platform (≥5 for sufficient)"
  - "Describes nature of work (production, scale, types of integrations)"
  - "Names current or recent employer and role scope"

3-5 anchors per question.

# RED FLAGS

Patterns indicating a weak or unverifiable claim:

  - "Vague years claim with no employer or scope context"
  - "Names a non-iPaaS tool (e.g., 'I used Zapier') as their primary platform"
  - "Cannot articulate what they actually built — generic phrases only"

2-3 red flags per question.

# FOLLOW-UPS (PROBES)

Sharp drill-downs that bluff-catch — specifics a real practitioner
would know. Examples for an iPaaS role:

  - "And how many total years across all your integration work, roughly?"
  - "Of MuleSoft, TIBCO, or Boomi specifically — which one have you
    used in production?"
  - "On that platform — roughly how many years hands-on, day-to-day building?"
  - "On MuleSoft, Mule 3 or 4? CloudHub or Runtime Fabric?"

0-3 follow-ups per question.

# RUBRIC

Frame the rubric as CLAIM-QUALITY bands, not design-quality:

  - `excellent`: states years + names platform + names years on platform
    + provides employer/scope context — concrete, falsifiable claims.
  - `meets_bar`: states plausible years and a platform with some scope.
    May lack precise years on the platform.
  - `below_bar`: refuses to engage, can't articulate any platform,
    or years contradict the claim implied by their disclosed background.

# EVALUATION HINT

Short hint to the screening engine — e.g.,
"Verify the claim and brief context — not depth. Concrete = names
platform, names years, names employer or scope."

# BUDGET ALLOTMENT (HARD CAP — SERVER-ENFORCED)

You have at most 3 minutes of mandatory budget across all behavioral
questions. Sum of estimated_minutes must NOT exceed 3.

# REMINDER

- `question_kind: "behavioral_star"` on every question.
- `is_mandatory: true` on every question.
- 1-2 questions, total ≤ 3 minutes.
- Anchors are claim shapes, NOT technical specifics.
- This is the verification gate before technical depth runs.
```

def _apply_mandatory_correction_in_position_order(
    *,
    questions: list[GeneratedQuestion],
    knockout_values: set[str],
) -> None:
    """Auto-correct is_mandatory in position order.

    For each knockout signal, the earliest question (by position) probing
    it claims the mandatory slot; later questions probing the same
    knockout are demoted to optional. Mutates `questions` in place.

    This is the post-merge pass for two-call generation — once behavioral
    and technical questions are concatenated in their final position order,
    this single pass enforces the "exactly one mandatory probe per knockout"
    rule across both kinds.
    """
    knockouts_covered: set[str] = set()
    for q in sorted(questions, key=lambda x: x.position):
        knockouts_in_q = set(q.signal_values) & knockout_values
        if not knockouts_in_q:
            continue
        unclaimed = knockouts_in_q - knockouts_covered
        if unclaimed:
            if not q.is_mandatory:
                logger.warning(
                    "question_bank.upgraded_to_mandatory",
                    signal_values=q.signal_values,
                    reason="earliest_knockout_question_must_be_mandatory",
                )
                q.is_mandatory = True
            knockouts_covered.update(unclaimed)
        else:
            if q.is_mandatory:
                logger.info(
                    "question_bank.demoted_to_optional",
                    signal_values=q.signal_values,
                    reason="duplicate_knockout_coverage",
                )
                q.is_mandatory = False
```

                eligible = _filter_behavioral_eligible(snapshot.signals)
                budget = BEHAVIORAL_BUDGET_MIN
                if prompt_name is None or not eligible:
                    # Nothing to regenerate — record skipped and exit clean
                    bank.generation_status_by_kind = {
                        **(bank.generation_status_by_kind or {}),
                        kind: "skipped_no_eligible_signals",
                    }
                    transition_to_reviewing_after_generation(
                        bank, user_id=started_by_uuid,
                    )
                    await db.commit()
                    publish_args = (job_id, stage_id, "reviewing")
                    return
            elif kind == "technical_depth":
                prompt_name = STAGE_TYPE_TO_PROMPT.get(stage.stage_type)
                eligible = snapshot.signals
                # Reduce by the OTHER kind's persisted total minutes
                from sqlalchemy import select as _select
                other_q = (
                    await db.execute(
                        _select(StageQuestion).where(
                            StageQuestion.bank_id == bank.id,
                            StageQuestion.question_kind == "behavioral_star",
                        )
                    )
                ).scalars().all()
                other_total = sum(float(q.estimated_minutes) for q in other_q)
                budget = max(1, int(stage.duration_minutes - other_total))
            else:
                raise RuntimeError(f"Unknown kind: {kind!r}")

            new_questions = await _generate_questions_for_kind(
                db,
                bank=bank,
                stage=stage,
                instance=instance,
                job=job,
                snapshot=snapshot,
                kind=kind,
                eligible_signals=eligible,
                budget_minutes=budget,
                prompt_name=prompt_name,
            )

            # Merge: keep existing questions of the OTHER kind (and recruiter rows of any kind),
            # append the regenerated ones, re-pack positions, run post-merge correction.
            from sqlalchemy import select as _select_q
            kept_q = (
                await db.execute(
                    _select_q(StageQuestion).where(StageQuestion.bank_id == bank.id)
                )
            ).scalars().all()

            # Convert kept rows to GeneratedQuestion-like for unified handling
            # — we actually just want positions: drop their slots, then re-pack.
            # Simpler: write the new questions and let write_generated_questions
            # handle the position re-pack (it preserves recruiter rows).
            await write_generated_questions(
                db, bank=bank, questions=new_questions, source="ai_generated",
            )

            # Re-fetch combined list for post-merge correction
            combined = await get_bank_questions(db, bank.id)
            from app.modules.question_bank.service import (
                _apply_mandatory_correction_in_position_order,
            )
            from app.modules.question_bank.schemas import GeneratedQuestion, QuestionRubric
            # Project ORM rows into GeneratedQuestion shape for the helper
            projected = [
                GeneratedQuestion(
                    position=q.position,
                    text=q.text,
                    signal_values=list(q.signal_values),
                    estimated_minutes=q.estimated_minutes,
                    is_mandatory=q.is_mandatory,
                    follow_ups=list(q.follow_ups),
                    positive_evidence=list(q.positive_evidence),
                    red_flags=list(q.red_flags),
                    rubric=QuestionRubric(**q.rubric),
                    evaluation_hint=q.evaluation_hint,
                    question_kind=q.question_kind,
                )
                for q in combined
            ]
            knockout_values = {
                s["value"] for s in snapshot.signals if s.get("knockout", False)
            }
            _apply_mandatory_correction_in_position_order(
                questions=projected, knockout_values=knockout_values,
            )
            # Apply corrections back to ORM rows
            by_position = {p.position: p for p in projected}
            for row in combined:
                if row.position in by_position:
                    row.is_mandatory = by_position[row.position].is_mandatory
            await db.flush()

            # Update per-kind status
            bank.generation_status_by_kind = {
                **(bank.generation_status_by_kind or {}),
                kind: "reviewing",
            }
            bank.pipeline_version_at_generation = instance.pipeline_version
            bank.stage_config_snapshot = {
                "signal_filter": stage.signal_filter,
                "difficulty": stage.difficulty,
            }
            bank.is_stale = False

            transition_to_reviewing_after_generation(bank, user_id=started_by_uuid)
            await log_event(
                db,
                tenant_id=tenant_uuid,
                actor_id=started_by_uuid,
                actor_email=None,
                action="question_bank.kind_regenerated",
                resource="stage_question_bank",
                resource_id=bank.id,
                payload={"kind": kind},
            )
            await db.commit()
            publish_args = (job_id, stage_id, "reviewing")
        except Exception as exc:
            logger.error(
                "question_bank.regenerate_kind_failed",
                bank_id=bank_id,
                kind=kind,
                error=str(exc)[:500],
                exc_info=True,
            )
            bank.generation_status_by_kind = {
                **(bank.generation_status_by_kind or {}),
                kind: "failed",
            }
            transition_to_failed(bank, error=f"{kind} regenerate failed: {str(exc)[:200]}")
            await db.commit()
            publish_args = (job_id, stage_id, "failed")

    if publish_args is not None:
        job_id_pub, stage_id_pub, new_status_pub = publish_args
        await pubsub.publish(
            pubsub.job_channel(job_id_pub),
            pubsub.Events.BANK_STATUS_CHANGED,
            {
                "job_id": str(job_id_pub),
                "bank_id": bank_id,
                "stage_id": str(stage_id_pub),
                "new_status": new_status_pub,
                "source": "actor",
            },
            correlation_id=effective_corr,
        )
```

        agent=agent, turn_id=turn_id, speaker_input=speaker_input,
    )
```

/**
 * useOnce: fires a callback exactly once on mount. Local helper —
 * the standard useEffect with an empty deps array would do this too
 * but adds a lint exception per render. Encapsulated.

    if (!fired.has(key)) {
      fired.add(key);
      callback();
    }
  }
}

Create `frontend/session/hooks/use-agent-state.ts`:

```ts
"use client";

import { useEffect, useState } from "react";
import { useRoomContext } from "@livekit/components-react";
import { ParticipantEvent, RoomEvent, type RemoteParticipant } from "livekit-client";

export type AgentState =
  | "initializing"
  | "listening"
  | "thinking"
  | "speaking"
  | "unknown";

/**
 * Subscribes to the agent participant's `lk.agent.state` attribute.
 * Returns the current state and a `hasSpoken` boolean that flips True
 * on the FIRST transition to "speaking" — used to dismiss the
 * IntroLoader once the agent has started speaking.
 *
 * See docs/superpowers/specs/2026-05-19-behavioral-layer-and-intro-design.md §3.
 */
export function useAgentState(): {
  state: AgentState;
  hasSpoken: boolean;
} {
  const room = useRoomContext();
  const [state, setState] = useState<AgentState>("unknown");
  const [hasSpoken, setHasSpoken] = useState(false);

  useEffect(() => {
    if (!room) return;

    function getAgentParticipant(): RemoteParticipant | undefined {
      // The agent identity prefix is set server-side. Match by attribute
      // role=agent — set on the agent's participant by LiveKit's framework.
      for (const p of room.remoteParticipants.values()) {
        if (p.attributes?.["lk.agent.state"] !== undefined) {
          return p;
        }
        if (p.attributes?.role === "agent") {
          return p;
        }
      }
      return undefined;
    }

    function readState(p: RemoteParticipant) {
      const raw = p.attributes?.["lk.agent.state"];
      const next = (raw ?? "unknown") as AgentState;
      setState(next);
      if (next === "speaking") {
        setHasSpoken(true);
      }
    }

    function handleAttributesChanged(
      _changed: Record<string, string>,
      participant: RemoteParticipant,
    ) {
      if (participant === getAgentParticipant()) {
        readState(participant);
      }
    }

    // Initial read (in case the agent attribute was already set when we mounted)
    const agent = getAgentParticipant();
    if (agent) readState(agent);

    // Subscribe to changes
    room.on(RoomEvent.ParticipantAttributesChanged, handleAttributesChanged);
    // Also re-evaluate when a participant connects (the agent joins after us)
    const handleParticipantConnected = (p: RemoteParticipant) => {
      readState(p);
    };
    room.on(RoomEvent.ParticipantConnected, handleParticipantConnected);

    return () => {
      room.off(RoomEvent.ParticipantAttributesChanged, handleAttributesChanged);
      room.off(RoomEvent.ParticipantConnected, handleParticipantConnected);
    };
  }, [room]);

  return { state, hasSpoken };
}
```
